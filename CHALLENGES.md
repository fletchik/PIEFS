# Трудности и столкновения при реализации — журнал

Этот файл фиксирует все нетривиальные проблемы, с которыми мы столкнулись в ходе реализации
и воспроизведения экспериментов из статьи. Предназначен для написания диплома и статьи.

---

## 1. `@torch.no_grad()` ломал `torch.autograd.grad` в `BasisFunction`

**Файл:** `src/trainer/sequential_trainer.py`, методы `_evaluate()` и `_compute_dirichlet_energy()`

**Симптом:** При вызове `_evaluate()` с декоратором `@torch.no_grad()` падал `RuntimeError`:
```
RuntimeError: element 0 of tensors does not require grad and does not have a grad_fn
```

**Причина:** `BasisFunction.forward()` делает `x_ = x.detach().requires_grad_(True)`, после чего
вызывает `torch.autograd.grad(phi, x_)`. Декоратор `@torch.no_grad()` глобально отключает построение
вычислительного графа, поэтому `phi` не имеет `grad_fn` → autograd.grad падает.

**Решение:** Убрали декоратор `@torch.no_grad()` с обоих методов. Параметры сети не накапливают
градиенты, потому что `x` детачится внутри самой `BasisFunction` (`x.detach()`).

**Урок для диплома:** При использовании `torch.autograd.grad` (не `.backward()`) нельзя помещать
вызывающий код под `torch.no_grad()`. Это тонкое отличие от обычного обучения.

## 2. `.numpy()` на тензоре с `requires_grad=True` в мультиклассовом eval

**Файл:** `src/trainer/sequential_trainer.py`, мультиклассовая ветка `_evaluate()`

**Симптом:**
```
RuntimeError: Can't call numpy() on Tensor that requires grad.
```

**Причина:** `head_out['probs']` для `MulticlassHead` возвращался с `requires_grad=True`
(потому что `@torch.no_grad()` был убран — см. проблему #1), а `.numpy()` требует отключённых
градиентов.

**Решение:** Добавили явное `.detach()` перед `.numpy()`:
```python
probs = out['head_out']['probs'].detach()
```

## 3. YAML-парсинг: `off` интерпретируется как `bool False`

**Файл:** `src/configs/train.yaml`, параметр `model.metric_type`

**Симптом:**
```
ValueError: Unknown metric_type 'False'
```

**Причина:** В спецификации YAML 1.1 слова `off`, `on`, `yes`, `no` — булевы значения.
Когда Hydra читает `metric_type: off`, Python получает `False`, а не строку `'off'`.

**Решение:** Цитирование строки: `metric_type: 'off'`. То же касается всех вызовов в
`reproduce_all.sh`: `model.metric_type='off'`.

**Урок для диплома:** YAML 1.1 vs YAML 1.2 — принципиальная разница. В Hydra используется
PyYAML (YAML 1.1), где `off/on/yes/no` — булевы. Это неочевидная ловушка при именовании
параметров.

## 4. Неправильная формула потери Дирихле: `∇φᵀ A ∇φ` вместо `‖A∇φ‖²`

**Файл:** `src/loss/spectral_loss.py`

**Симптом:** Энергии Дирихле при использовании `lambda_u_sparse` были отрицательными:
`eigenvalue proxy = -1.2e+22`.

**Причина:** Была реализована билинейная форма `∇φᵀ A ∇φ`, которая для несимметричной
матрицы A = ΛU (где U — ортогональная матрица вращения) может давать отрицательные
значения (U не сохраняет знак квадратичной формы).

Статья eq. 5-6 описывает другую формулу: `‖A∇f‖²`, то есть сначала применяем A к градиенту,
потом берём норму. Это всегда неотрицательно.

**Решение:**
```python
# Было (неверно):
Ag = torch.bmm(A, grad.unsqueeze(-1)).squeeze(-1)
loss = (grad * Ag).sum(dim=1).mean()  # ∇φᵀ A ∇φ — может быть < 0

# Стало (верно):
Ag = torch.bmm(A, grad.unsqueeze(-1)).squeeze(-1)
loss = (Ag ** 2).sum(dim=1).mean()   # ‖A∇φ‖² — всегда ≥ 0
```

**Урок для диплома:** Для несимметричной A формула Дирихле `‖A∇f‖²` и `∇fᵀ Aᵀ A ∇f`
эквивалентны (Aᵀ A — симметричная PSD матрица), но `∇fᵀ A ∇f` — это другая формула и
принципиально не то, что имеется в виду в статье.

## 5. Синтаксис Hydra: `model/metric=diag` vs `model.metric_type=diag`

параметр `model.metric_type`.

**Решение:** Заменили на `model.metric_type=diag` (dot-notation для обычного параметра).

## 6. Константные собственные функции на единичной окружности нарушали проверку порядка

**Файл:** `scripts/verify_circle.py`

**Симптом:** Check 4 (корреляция с cos/sin на окружности) падал с ошибкой порядка:
```
FAIL: ordering_satisfied=False — энергии [0.37, 0.0, ...]
```

**Причина:** φ_1 и φ_4 оказались константными функциями (энергия Дирихле ≈ 0 < 0.01).
Проверка монотонности включала их в сравнение: 0.37 > 0.0 → нарушение порядка.

**Решение:** Изменили проверку — исключаем из ordering-сравнения функции с энергией < 0.01.
Для них выводится `SKIP (constant eigenfunction)`.

**Урок для диплома:** Последовательное обучение может застрять в тривиальном минимуме
(φ ≡ const), особенно если предыдущие функции уже объяснили большую часть вариации данных.
Константная функция ортогональна любой функции с нулевым средним, поэтому потеря Грама
не штрафует её.

## 7. Энергии Дирихле не монотонно возрастают (`ordering_satisfied=False`)

**Файл:** `src/trainer/sequential_trainer.py`, логика после заморозки функции

**Наблюдение:** В большинстве прогонов Groups A–F флаг `ordering_satisfied=False`.
Например, two_moon_off_s42:
```
φ_1: E=0.030  ordering=True
φ_2: E=0.003  ordering=False  ← регресс
φ_3: E=0.305  ordering=False
...
```

**Причина:** Мы не гарантируем монотонность энергий — это свойство точных собственных функций
оператора Дирихле (λ₁ ≤ λ₂ ≤ ...), а не приближённых нейросетевых. Так как оптимизация
приближённая и с несколькими конкурирующими слагаемыми потерь, порядок не соблюдается.

**Статус:** Известное ограничение метода. Для диплома — честно задокументировать как отклонение
приближённого решения от теоретического.

## 11. gram_error не сходится к 0.01 — эффект накопления

**Наблюдение:** Финальная метрика `‖C_K − I_K‖_F` составляет 0.15–0.45 при цели 0.01.

**Причина (математическая):**
Норма Фробениуса K×K матрицы накапливает ошибку от всех K(K−1)/2 попарных перекрытий:
```
‖C − I‖_F ≈ ε_pair · √(K(K−1))
```
При K=6 и ε_pair ≈ 0.04: `‖C−I‖_F ≈ 0.04 · √30 ≈ 0.22` — что и наблюдается.

**Причины (алгоритмические):**
- Замороженные функции не обновляются → ранние перекрытия фиксируются навсегда
- Мягкий штраф (gram loss) конкурирует с L_task и L_mde
- 10k шагов на функцию недостаточно (статья: ~2 часа на функцию)

**Ответ статьи:** Статья не указывает целевое значение gram_error явно. Она описывает только
иерархию потерь (eq. 10) и упоминает, что T_orth задаётся на этапе предобучения (GL pretraining).
Конвергенция достигается за счёт большого числа итераций + хорошей инициализации.

**Предполагаемые улучшения:**
1. Gram-Schmidt дефляция — ортогональность по конструкции, а не штраф
2. Увеличить `w_gram` (с 1.0 до 10.0) и/или шаги (с 10k до 20k)
3. Graph Laplacian предобучение для хорошей инициализации

## 13. `metric_type='off'` = нет метрики, лучше чем `diag` на Circles

**Наблюдение:** На датасете Circles `metric_type='off'` (стандартный ‖∇φ‖²) даёт val_acc=0.979,

**Следствие для статьи:** Усложнение метрики (lambda_u_sparse) не всегда помогает и может
вредить на простых геометриях.

## 14. `ordering_satisfied` — неправильная метрика для нейросетевых приближений

**Текущий мониторинг:** Логируем `ordering_satisfied = E_k ≤ E_{k+1}`.

**Проблема:** Это свойство точных собственных функций. Для нейросетевых приближений с
несколькими конкурирующими слагаемыми потерь — нереалистичная метрика.

**Рекомендация:** Заменить/дополнить на `max_pairwise_overlap = max|C[i,j]|` для i≠j —
это более честная метрика того, насколько функции ортогональны попарно.

## 15. Нормализация потери Дирихле для тождественной метрики (metric='off')

**Файл:** `src/loss/spectral_loss.py`, строка ~99

**Симптом:** При сравнении metric='off' vs 'diag', функция 'off' имела аномально малое Dirichlet слагаемое потери.

**Причина:** Для тождественной метрики использовался `.mean()`, который делит на B×d, тогда как для диагональной и полной метрик — `.sum(dim=1).mean()` (делит только на B). Разница в d раз:
```python
# Было (неверно):
loss_dirichlet = (grad_phi_k ** 2).mean()  # делит на B×d

# Стало (верно):
loss_dirichlet = (grad_phi_k ** 2).sum(dim=1).mean()  # делит на B
```

**Последствие:** Для HTRU2 (d=8): MDE для 'off' была в 8 раз слабее чем для 'diag'.
Для MNIST (d=784): в 784 раза слабее! Сравнение метрик в ночных экспериментах некорректно.

**Урок:** При реализации норм векторов/матриц всегда явно указывать оси: `.sum(dim=1).mean()` vs `.mean()` — принципиально разные величины при d > 1.

## 16. Неверные метки для точек wide normal augmentation

**Файл:** `src/trainer/sequential_trainer.py`, метод `_augment_batch()`

**Симптом:** При использовании `wide_normal_fraction > 0` на мультиклассовых задачах — смещение классификационной головы в сторону класса 0.

**Причина:**
```python
# Было (неверно): все "широкие" точки получают метку 0
wide_labels = torch.zeros(n_wide, dtype=y.dtype, device=y.device)
```

Для 10-классного MNIST все n_wide точек имели метку 0, что обучало голову предсказывать класс 0 для "бесконечно далёких" точек — нежелательное поведение.

**Решение:**
```python
# Стало (верно): случайные метки из реального батча
rand_idx = torch.randint(0, n_real, (n_wide,), device=y.device)
wide_labels = y[:n_real][rand_idx]
```

"Широкие" точки по замыслу статьи должны регуляризовать Dirichlet-член (гладкость в пространстве), но не влиять на задачу классификации — случайные метки обеспечивают нейтральность.

## 17. PINN обучается на единичных векторах, но apply_to() подаёт произвольные w=Λ∇φ

**Файл:** `src/model/metric/lambda_u_pinn.py`, метод `apply_to()`

**Симптом:** При больших значениях λ (диагональный масштаб) или больших ||∇φ|| вектор w=Λ∇φ имеет норму 2–8+ (для HTRU2) или 10–50+ (для MNIST), тогда как PINN обучался на единичных векторах.

**Причина:** В `pretrain()` использовалось `v0 = F.normalize(v0, dim=-1)`, но `apply_to()` подавал w = lam * grad без нормализации. Tanh-активации PINN насыщаются при больших входах.

**Решение:** Нормализация и обратное масштабирование в `apply_to()`:
```python
norms = w.norm(dim=-1, keepdim=True).clamp(min=1e-8)
w_unit = w / norms
u_w_unit = self._pinn(omega_v, w_unit)
return u_w_unit * norms  # U·w ≈ ||w||·U·(w/||w||)
```

Это корректно для точного вращения (U линейно), и предотвращает насыщение Tanh.
